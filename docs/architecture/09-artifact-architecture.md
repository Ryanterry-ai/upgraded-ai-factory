# Artifact Architecture

## Overview
Artifacts are the persistent outputs generated by agents, workflows, and factories. This document defines artifact ownership, versioning, storage, and lifecycle management.

## Ownership Rules

### Primary Ownership
- Each artifact has exactly one owning agent
- Owner is responsible for creation, updates, and deletion
- Owner must validate artifact before marking complete

### Transfer of Ownership
- Artifacts can be transferred between agents via Coordinator
- Transfer requires audit trail in Project Memory
- Previous owner loses write access after transfer

### Read Access
- All agents can read any artifact in their project
- Cross-project access requires Coordinator approval
- External access requires CEO approval

## Versioning Strategy

### Blueprint Versioning
```yaml
version: 1.0.0
history:
  - version: 1.0.0
    created_at: "2026-01-15T10:00:00Z"
    created_by: "blueprint-generator-agent"
    changes: ["Initial blueprint generation"]
  - version: 1.1.0
    created_at: "2026-01-15T10:05:00Z"
    created_by: "blueprint-generator-agent"
    changes: ["Added missing responsive breakpoints"]
```

### Source Code Versioning
- All source code changes tracked via Git
- Each agent commit includes agent identifier in commit message
- Branch strategy: `main` (production), `dev` (development), `feature/*` (agent work)

### Artifact Version Increment Rules
- Major (X.0.0): Breaking changes to schema or structure
- Minor (0.X.0): New sections or capabilities added
- Patch (0.0.X): Bug fixes or corrections

- **Working Memory (Redis)**: Volatile context, session data

## Lifecycle Management

### Creation
1. Agent creates artifact
2. Agent validates artifact against schema
3. Agent registers artifact in Project Memory
4. Agent notifies Coordinator of completion

### Update
1. Agent reads existing artifact
2. Agent modifies artifact
3. Agent increments version
4. Agent updates Project Memory record
5. Agent notifies affected agents

### Deletion
1. Coordinator approves deletion request
2. Agent archives artifact (optional)
3. Agent removes artifact from storage
4. Agent removes Project Memory record
5. Coordinator confirms deletion

### Archival
1. Coordinator identifies artifacts past retention period
2. Coordinator moves artifacts to archive storage
3. Coordinator updates Project Memory with archive reference
4. Archived artifacts can be restored on demand

| Configuration | Permanent | Permanent | Manual only |
| Source Code | Permanent | Permanent | Manual only |
| Screenshots | 30 days | 90 days | Auto-delete |
| Build Logs | 30 days | 90 days | Auto-delete |
| Test Results | 30 days | 90 days | Auto-delete |
| Reviews | 90 days | 365 days | Auto-delete |
| Working Memory | TTL (ms-hours) | N/A | Auto-expire |

## Integrity Checks

### Checksums
- All artifacts include SHA-256 checksums
- Checksums stored in Project Memory
- Verified on read and before deployment

### Validation
- Schema validation on creation
- Type checking on source code
- Build validation on compilation
- Visual validation on screenshots

### Recovery
- Artifacts restored from backup on corruption
- Checksum verified after restoration
- Affected agents notified of recovery
